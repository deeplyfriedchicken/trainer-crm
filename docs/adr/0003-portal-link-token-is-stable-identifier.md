# Portal link token is a stable identifier, not an expiring credential

The Client Portal URL (`/client/[token]`) embeds an AES-256-GCM encrypted trainee ID. Early versions of the encoder included a 30-day TTL inside the encrypted payload and rejected expired tokens by returning `null` from `decryptUserId`, which caused pages to render a 404 once the TTL elapsed — even when the trainee was on a correct URL and could have re-entered their PIN. This conflated two distinct concerns: the URL as a non-enumerable identity carrier, and the URL as an expiring auth credential. The actual auth boundary is the trainee's 6-digit PIN plus the 7-day `client_session` JWT cookie — those layers gate every authenticated action; the URL token alone never authorises anything. We decided to remove the TTL from the token entirely: any token whose GCM auth tag verifies returns its trainee ID, and the existing PIN modal handles the "no session" case automatically. Alternatives considered: (a) keep the TTL but render the PIN modal on expiry instead of 404 — adds a second decode path that contradicts the identifier framing and leaves a half-meaningful field; (b) extend the TTL to ~10 years — kicks the problem down the road and leaves a dead field future maintainers may resurrect. The trade-off accepted: leaked portal links remain useful indefinitely, but a holder still needs the PIN to access anything, and trainers can reset a trainee's PIN at any time to revoke access regardless of how many copies of the URL are floating around. The decoder retains a lenient `indexOf(":")` split so previously-issued `userId:expiresAt` tokens continue to decode for the lifetime of every existing PWA install.
